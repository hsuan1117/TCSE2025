Building a High-Throughput IoT Health Monitoring System with MQTT and Redis

Hung-Wei Cheng - National Yang Ming Chiao Tung University

Jui-Hsuan Chang - National TsingHua University

Challenge: How do we handle real-time physiological data from thousands of IoT devices without compromising performance?

Node.js MQTT Redis PostgreSQL Prisma ORM

The Scaling Challenge

Our IoT health monitoring system collects critical data:

  • Heart rate measurements
  • Respiratory rate monitoring
  • Sleep quality analysis
  • Movement and position tracking

Initial Architecture Problems:

  • Direct database writes from MQTT messages
  • Emergency queries taking too long (scanning recent time windows)
  • Task queues building up under load
  • System instability with increasing device count

Original System Architecture

IoT Devices
→
MQTT Broker
→
PostgreSQL

Key Issues:

  • Tight coupling between message ingestion and database writes
  • No buffering mechanism for burst traffic
  • Emergency detection queries directly hitting the database
  • Linear performance degradation with device growth

Result: System became unusable under high load, threatening patient safety in emergency scenarios

Optimized Architecture with Redis

IoT Devices
→
MQTT Broker
→
Redis Buffer
→
Background Processor
→
PostgreSQL

Key Improvements:

  • Redis as high-speed in-memory buffer
  • Decoupled producer-consumer model
  • Asynchronous batch processing
  • TTL-based data expiration for recent queries

Redis Implementation Strategy

Data Structure Design:

  • mqtt_messages - Redis List for incoming messages
  • d_heartRate_deviceId - Sorted sets for vital signs
  • d_breath_deviceId - Time-series data with TTL
  • Automatic expiration after 5 minutes

Emergency Detection via Redis:

  • Uses zRemRangeByScore for sliding window
  • Real-time averaging of recent measurements
  • Status levels: NORMAL, WARNING, ALERT, OFFLINE
  • Triggers: breath < 10, heartRate < 40, or no data

Benefit: Emergency queries now use Redis instead of PostgreSQL, reducing latency from seconds to milliseconds

Trade-offs and Lessons Learned

Benefits Achieved:

  • Massively improved scalability and responsiveness
  • Decoupled architecture enables independent scaling
  • Real-time emergency detection restored
  • Production-ready for healthcare deployments

Trade-offs to Consider:

  • Complexity: Additional layer requires synchronization
  • Data Risk: Potential loss if Redis crashes before processing
  • Operational Overhead: Monitoring dual storage systems
  • TTL Management: Critical data might expire if processing delays

Verdict: Trade-offs are acceptable given the critical nature of health monitoring and achieved performance gains

Future Enhancements

Planned Improvements:

  • Redis Streams/Pub-Sub for event-driven pipelines
  • Redis persistence (RDB/AOF) to mitigate data loss
  • Multiple worker instances with load balancing
  • Real-time monitoring dashboard for ops teams
  • Advanced analytics with time-series databases

Key Takeaway

When building real-time IoT systems, consider buffering layers early. The complexity of adding Redis was worth the dramatic improvements in performance and reliability.
